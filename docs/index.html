01 · How it works

A static audit of the disk, before the next turn boots.

/context shows the current conversation. /pms-inspector reads your ~/.claude/ config and predicts what the next session will load — no LLM calls, no writes, no network.

  1. 1

    Discover

    Walk enabledPlugins and every MCP server, skill, agent, and hook on disk.

  2. 2

    Measure

    Count bytes for frontmatter + description; convert at 4 chars ≈ 1 token.

  3. 3

    Classify

    Apply skillListingBudgetFraction and skillListingMaxDescChars. Four states, no fuzz.

  4. 4

    Report

    Human-aligned tables (CJK-aware widths) or --json for pipelines.

  5. 5

    Suggest

    Concrete knob values with token deltas — copy-paste into settings.json.

02 · The four load states

Every skill lands in exactly one square.

Each square is a Pantone-style chip because PMS is also the Pantone Matching System — and this whole tool is about matching what you shipped against what actually got loaded.

PMS 5773 CFULL

Description reaches the model intact — the ideal state for a skill you actually want triggered.

PMS 138 CDESC-TRUNCATED

Capped by skillListingMaxDescChars (default 1536). The tail is silently cut.

PMS 165 CBUDGET-COMPRESSED

Squeezed by the global skillListingBudgetFraction. Text remains, but shorter than the file.

PMS 179 CNAME-ONLY

Description dropped entirely — the model sees only the slug and can’t reason about when to invoke it.

03 · Reference case · 318 skills · 200 k window

Two settings. Roughly 17 k tokens between them.

Before — default budget 0.01

318 / 318 name-only

~2 000 tokens spent, but the model doesn’t know why any skill exists. Half your invocations miss.

After — budget raised to 0.15

318 / 318 full

~18 900 tokens (9.4 % of the window). Every skill’s trigger sentence reaches the model. Invocations start landing.

04 · What the tool actually inspects

Three knobs. Four states. One truth.

skillListingBudgetFraction
Fraction of the context window Claude Code reserves for skill descriptions. Default 0.01. On a busy setup this is the single knob that flips everything to name-only.
skillListingMaxDescChars
Hard cap per description before it’s marked desc-truncated. Default 1536. Rarely the bottleneck — usually the fraction bites first.
--ctx <tokens>
Override the assumed context window (default 200000). Handy when you’re modelling Haiku or an older tier.
--json
Machine-parseable output. All four load states, byte / token counts, and per-skill breakdown — pipe into whatever you already use.
--verbose
Per-skill breakdown in the human report. Useful when one skill is disproportionately heavy.
--lang <code>
Force output language: en · zh-CN · zh-TW · ja · ko · fr · de · es. Auto-detected from settings.json otherwise.
