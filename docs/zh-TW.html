Plugin · MCP · Skill 檢查器

看清你的工作階段究竟載入了什麼。

一個零依賴的 Claude Code 外掛，讀取你磁碟上的設定，為每一個 plugin、MCP、skill、agent、hook 列出位元組 / token 帳本 —— 並告訴你哪一個參數決定了 300 個 skill 的開銷是 2k 還是 20k tokens / turn。

01 · 運作原理

在下一個 turn 開始之前，先做一次磁碟靜態審計。

/context 看的是目前工作階段。/pms-inspector 讀你 ~/.claude/ 裡的設定，預測下一個工作階段會載入什麼 —— 不呼叫 LLM、不寫磁碟、不連線。

  1. 1

    發現

    走訪 enabledPlugins 與磁碟上每一個 MCP、skill、agent、hook。

  2. 2

    度量

    對 frontmatter + description 計位元組，按 4 字元 ≈ 1 token 換算。

  3. 3

    分類

    套用 skillListingBudgetFraction 與 skillListingMaxDescChars。四種狀態，界線清楚。

  4. 4

    報告

    人眼對齊的表格（含 CJK 視覺寬度）或 --json 供流水線使用。

  5. 5

    建議

    給出具體的參數值 + token 差額 —— 複製貼上至 settings.json。

02 · 四種載入狀態

每一個 skill 都必然落到其中一格。

這四張色卡採用 Pantone 風格 —— 因為 PMS 剛好也是 Pantone Matching System。這個工具做的事，本質上就是把你發布的和真正被載入的做一次校色。

PMS 5773 CFULL

描述完整送達模型 —— 你希望被觸發的 skill 就該落在這一格。

PMS 138 CDESC-TRUNCATED

被 skillListingMaxDescChars（預設 1536）截斷。尾端被靜默切掉。

PMS 165 CBUDGET-COMPRESSED

被全域 skillListingBudgetFraction 擠壓。文字還在，但比原檔短。

PMS 179 CNAME-ONLY

描述完全遭丟棄 —— 模型只看到 slug，無從判斷何時應該呼叫它。

03 · 參考案例 · 318 skills · 200k 視窗

兩組設定。差距大約 17k tokens。

之前 —— 預設 budget 0.01

318 / 318 name-only

只花了約 2 000 tokens，但模型不知道任何 skill 存在的理由。一半的呼叫都打不中。

之後 —— budget 調到 0.15

318 / 318 full

約 18 900 tokens（視窗的 9.4%）。每一個 skill 的觸發描述都送達模型。呼叫開始命中。

04 · 工具實際檢查的是什麼

三個旋鈕。四種狀態。一份真相。

skillListingBudgetFraction
Claude Code 為 skill 描述保留的上下文比例。預設 0.01。skill 一多，這個參數會讓所有 skill 一起掉到 name-only。
skillListingMaxDescChars
單條描述在被標記為 desc-truncated 之前的硬上限。預設 1536。很少是瓶頸 —— 通常是 fraction 先咬到。
--ctx <tokens>
覆寫假定的上下文視窗（預設 200000）。給 Haiku 或舊 tier 建模時用得上。
--json
機器可解析輸出。四種載入狀態、位元組 / token 計數、每個 skill 明細 —— 直接進你的 pipeline。
--verbose
人類報告裡附帶每個 skill 的明細。適合排查某一個 skill 佔用異常大的情況。
--lang <code>
強制輸出語言：en · zh-CN · zh-TW · ja · ko · fr · de · es。不指定則從 settings.json 自動偵測。
