01 · Funktionsweise

Ein statisches Audit der Festplatte, bevor der nächste Turn startet.

/context zeigt die aktuelle Unterhaltung. /pms-inspector liest Ihre Konfiguration in ~/.claude/ und sagt voraus, was die nächste Sitzung laden wird — kein LLM-Aufruf, kein Schreiben, kein Netzwerk.

  1. 1

    Erfassen

    Über enabledPlugins und jeden MCP-Server, jedes Skill, jeden Agent und jeden Hook auf der Platte laufen.

  2. 2

    Messen

    Bytes für Frontmatter + Beschreibung zählen; mit 4 Zeichen ≈ 1 Token umrechnen.

  3. 3

    Klassifizieren

    skillListingBudgetFraction und skillListingMaxDescChars anwenden. Vier Zustände, keine Unschärfe.

  4. 4

    Berichten

    Menschenlesbare, ausgerichtete Tabellen (CJK-Breite bewusst) oder --json für Pipelines.

  5. 5

    Vorschlagen

    Konkrete Reglerwerte mit Token-Deltas — direkt in settings.json einfügen.

02 · Die vier Ladezustände

Jedes Skill landet in genau einem Feld.

Jedes Feld ist ein Chip im Pantone-Stil, weil PMS auch Pantone Matching System heißt — und dieses Werkzeug macht nichts anderes, als das, was Sie ausgeliefert haben, mit dem abzugleichen, was tatsächlich geladen wurde.

PMS 5773 CFULL

Die Beschreibung erreicht das Modell unversehrt — der Idealzustand für ein Skill, das Sie wirklich ausgelöst haben wollen.

PMS 138 CDESC-TRUNCATED

Durch skillListingMaxDescChars (Standard 1536) gedeckelt. Das Ende wird still abgeschnitten.

PMS 165 CBUDGET-COMPRESSED

Vom globalen skillListingBudgetFraction zusammengequetscht. Text bleibt, aber kürzer als die Datei.

PMS 179 CNAME-ONLY

Beschreibung komplett verworfen — das Modell sieht nur den Slug und kann nicht schlussfolgern, wann es zu triggern ist.

03 · Referenzfall · 318 Skills · 200-k-Fenster

Zwei Einstellungen. Etwa 17k Tokens dazwischen.

Vorher — Standard-Budget 0,01

318 / 318 name-only

~2 000 Tokens verbraucht, aber das Modell weiß nicht, warum es irgendein Skill gibt. Die Hälfte Ihrer Aufrufe geht daneben.

Nachher — Budget auf 0,15

318 / 318 full

~18 900 Tokens (9,4 % des Fensters). Der Trigger-Satz jedes Skills erreicht das Modell. Aufrufe landen wieder.

before · fraction 0.01
skills     318 discovered
  name-only      318   ← 100 %
  full             0

model sees:  318 slugs, zero context
est. cost:   ~ 2.0 k tokens/turn
after · fraction 0.15
skills     318 discovered
  full           318   ← 100 %
  name-only        0

model sees:  318 slugs + full descriptions
est. cost:  ~ 18.9 k tokens/turn  (9.4 % of window)

04 · Was das Werkzeug tatsächlich prüft

Drei Regler. Vier Zustände. Eine Wahrheit.

skillListingBudgetFraction
Anteil des Kontextfensters, den Claude Code für Skill-Beschreibungen reserviert. Standard 0.01. In einer vollen Konfiguration ist das der eine Regler, der alles auf name-only kippt.
skillListingMaxDescChars
Hartes Limit pro Beschreibung, bevor sie als desc-truncated markiert wird. Standard 1536. Selten der Engpass — meist beißt zuerst der Fraction-Regler.
--ctx <tokens>
Angenommenes Kontextfenster überschreiben (Standard 200000). Praktisch, wenn Sie Haiku oder eine ältere Klasse modellieren.
--json
Maschinenlesbare Ausgabe. Alle vier Zustände, Byte-/Token-Zählungen und Aufschlüsselung pro Skill — in jede Pipeline pipebar.
--verbose
Aufschlüsselung pro Skill im menschlichen Report. Nützlich, wenn ein Skill überproportional schwer wiegt.
--lang <code>
Ausgabesprache erzwingen: en · zh-CN · zh-TW · ja · ko · fr · de · es. Andernfalls automatisch aus settings.json erkannt.
