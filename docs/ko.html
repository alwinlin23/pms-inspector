01 · 작동 방식

다음 턴이 시작하기 전에, 디스크를 정적으로 감사합니다.

/context 는 현재 대화를 보여줍니다. /pms-inspector 는 ~/.claude/ 설정을 읽고 다음 세션이 무엇을 로드할지 예측합니다 —— LLM 호출 없음, 쓰기 없음, 네트워크 없음.

  1. 1

    발견

    enabledPlugins 와 디스크의 모든 MCP / skill / agent / hook 을 순회.

  2. 2

    측정

    frontmatter + description 의 바이트를 세고, 4 자 ≒ 1 토큰으로 환산.

  3. 3

    분류

    skillListingBudgetFraction 과 skillListingMaxDescChars 적용. 4 가지 상태, 모호함 없음.

  4. 4

    리포트

    사람이 보기 좋은 정렬 테이블 (CJK 너비 대응) 또는 파이프라인용 --json.

  5. 5

    제안

    구체적인 노브 값과 토큰 델타 —— settings.json 에 그대로 붙여넣기 가능.

02 · 4가지 로드 상태

모든 skill 은 정확히 한 칸에 떨어집니다.

각 칸이 Pantone 스타일 칩인 이유는 PMS 가 Pantone Matching System 이기도 하기 때문 —— 그리고 이 도구가 하는 일은 결국 당신이 배포한 것과 실제로 로드된 것의 컬러 매칭입니다.

PMS 5773 CFULL

description 이 모델에 완전히 전달됨 —— 실제로 트리거되기를 원하는 skill 이 있어야 할 자리.

PMS 138 CDESC-TRUNCATED

skillListingMaxDescChars (기본 1536) 로 잘림. 뒷부분이 조용히 잘립니다.

PMS 165 CBUDGET-COMPRESSED

글로벌 skillListingBudgetFraction 에 의해 압축됨. 텍스트는 남지만 원본보다 짧습니다.

PMS 179 CNAME-ONLY

description 이 완전히 사라짐 —— 모델은 slug 만 보고 언제 호출해야 할지 판단할 수 없습니다.

03 · 참조 케이스 · 318 skills · 200k 윈도우

설정 2 개. 그 사이의 차이는 약 17k 토큰.

이전 —— 기본 budget 0.01

318 / 318 name-only

약 2 000 토큰 소비. 하지만 모델은 어떤 skill 이 왜 존재하는지 모릅니다. 호출의 절반은 빗나갑니다.

이후 —— budget 을 0.15 로

318 / 318 full

약 18 900 토큰 (윈도우의 9.4%). 모든 skill 의 트리거 설명이 모델에 도달합니다. 호출이 맞아 들어가기 시작합니다.

04 · 이 도구가 실제로 검사하는 것

노브 3 개. 상태 4 가지. 진실 1 가지.

skillListingBudgetFraction
Claude Code 가 skill description 에 예약하는 컨텍스트 비율. 기본 0.01. skill 이 많은 설정에서는 이 하나로 모두 name-only 로 떨어집니다.
skillListingMaxDescChars
description 이 desc-truncated 로 표시되기 전의 하드 상한. 기본 1536. 병목이 되는 경우는 드묾 —— 보통 fraction 이 먼저 물립니다.
--ctx <tokens>
가정하는 컨텍스트 윈도우를 재정의 (기본 200000). Haiku 나 구세대 tier 를 모델링할 때 유용.
--json
기계가 읽을 수 있는 출력. 4 가지 로드 상태, 바이트 / 토큰 카운트, skill 별 세부 —— 사용 중인 파이프라인에 그대로 파이프.
--verbose
사람 대상 리포트에 skill 별 세부 표시. 특정 skill 이 유독 무거울 때 유용.
--lang <code>
출력 언어 강제: en · zh-CN · zh-TW · ja · ko · fr · de · es. 지정하지 않으면 settings.json 에서 자동 감지.
